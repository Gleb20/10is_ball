# Версионирование Tab-10

Формат: **`a.b.c`** (три неотрицательных целых числа).

## Семантика цифр

| Цифра | Когда увеличивать | Примеры |
|-------|-------------------|---------|
| **a** | Новый **флоу**: новые экраны, новый функционал, новые пользовательские сценарии end-to-end | Первый турнирный движок, раздел «Команды», judge mode |
| **b** | Новый функционал **внутри существующих** экранов/флоу или переписывание поведения под новую задачу без нового маршрута | Фильтры в истории, Undo в судействе, расширение формы матча |
| **c** | Исправление багов, UX-доработки, мелкие корректировки без нового поведения | Фикс валидации, отступы, тексты ошибок, стабильность |

## Правило сброса

При увеличении **левой** цифры все цифры **справа** обнуляются:

- `1.1.1` + **b** (функционал в существующих экранах) → **`1.2.0`**
- `1.2.3` + **a** (новый флоу) → **`2.0.0`**
- `2.0.5` + **a** → **`3.0.0`**
- `1.0.0` + **c** → **`1.0.1`**

> Пример из обсуждения «новая фича / глобальная переработка с 1.1.1» по правилу сброса даёт **`2.0.0`**, не `2.2.0`.

Phase 10 (UI polish): выравнивание bottom bar под [`05_UX_FLOWS.md`](requirements/05_UX_FLOWS.md) §1 считается **b** (переписывание shell под уже заявленный MVP), не **a** — см. ADR D5 в [`DECISIONS.md`](DECISIONS.md).

## Где фиксируется версия

- Корневой [`package.json`](../package.json) — поле `version` (источник для релиза)
- [`CHANGELOG.md`](../CHANGELOG.md) — пользовательский changelog по версиям
- [`docs/CHANGELOG_DEV.md`](CHANGELOG_DEV.md) — технический журнал шагов разработки
- Git-теги: `v{a.b.c}` (например `v1.2.0`)

## Связь с коммитами

В каждом коммите указывать, **затрагивает ли** изменение версию и какую цифру планируется поднять при следующем релизе (см. `.cursor/rules/git-commits.mdc`).
