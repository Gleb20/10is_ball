# Tab-10 MVP — автономный пакет требований для Cursor

Статус: Ready for implementation planning v2.0  
Дата: 2026-07-20  
Язык продукта: русский  
Подход к разработке: Test-Driven Development

## 1. Назначение пакета

Этот архив — единый источник требований для первой автономной версии Tab-10: закрытого mobile-first веб-сервиса для организации матчей и турниров по настольному теннису, ведения счёта, статистики, рейтинга и команд.

### Факты
- Первая версия является самостоятельным приложением.
- Пользователи создаются вручную через локальную теннисную админку.
- Внешних контуров учётных записей, межсервисных доступов и федеративного входа нет.
- Все функциональные изменения разрабатываются через цикл `Red → Green → Refactor`.

### Допущения
- MVP разворачивается для одной закрытой группы пользователей Moscow transport.
- Одна инсталляция обслуживает одну организацию; отдельная сущность организации не нужна.
- Сервер: 2 CPU, 2 GB RAM, 30 GB NVMe.
- Ожидается около 250 пользователей и 8–10 параллельных матчей.

### Рекомендации
- Не добавлять в MVP архитектурные заделы, которые не нужны текущей инсталляции.
- Не писать production-код до появления падающего теста на новое поведение.
- Использовать `04_PRD.md` как источник функциональных требований, а `11_ACCEPTANCE_TEST_CATALOG.md` — как источник проверяемых сценариев.

## 4. Граница MVP

### Входит
- локальный вход по email и паролю;
- локальная теннисная админка;
- ручное создание, блокировка и восстановление доступа пользователей;
- главная, профиль, рейтинги, история;
- матчи `1v1` и `2v2`;
- турниры single elimination и double elimination;
- judge mode, Undo и аварийная коррекция;
- команды, приглашения и внутрисервисные уведомления;
- онбординг, FAQ и feedback;
- пустые состояния;
- TDD, автоматические unit/integration/API/E2E тесты.

### Не входит
- самостоятельная регистрация;
- регистрационные заявки;
- подтверждение email;
- отправка писем;
- вход через сторонних провайдеров;
- единая учётка для нескольких продуктов;
- сервисные гранты и межсервисные роли;
- несколько организаций;
- микросервисы;
- WebSocket;
- round robin;
- несколько партий в матче;
- публичный доступ;
- редактирование завершённых результатов;
- ачивки;
- сохранение черновиков.

## 5. Критическое правило для Cursor

Любая задача должна содержать:
1. идентификатор требования;
2. тестовые сценарии до реализации;
3. минимальную реализацию для прохождения тестов;
4. рефакторинг без изменения поведения;
5. обновление traceability matrix;
6. полный зелёный прогон CI.
