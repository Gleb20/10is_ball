# Instructions for Cursor

## 1. Перед началом

Прочитать документы в порядке из `00_README.md`. Не генерировать код после чтения только одного PRD.

## 2. Архитектурные ограничения

- автономное приложение;
- modular monolith;
- одна PostgreSQL;
- локальные users/sessions;
- только роли `admin/user` на системном уровне;
- никаких внешних контуров входа;
- никаких микросервисов;
- никаких скрытых future abstractions без текущего use case.

## 3. Рабочий режим

Для каждой задачи:
1. назвать requirement ID;
2. предложить тесты;
3. написать тесты и показать, почему они падают;
4. реализовать минимальный код;
5. запустить релевантные тесты;
6. рефакторить;
7. запустить полный suite;
8. обновить docs/schema/traceability.

Не создавать большие партии кода без промежуточного зелёного состояния.

## 4. При неоднозначности

- факты из PRD обязательны;
- допущения должны быть явно названы;
- предложить 2–3 варианта;
- предпочесть самый простой вариант, не нарушающий acceptance tests;
- не добавлять scope самостоятельно.

## 5. Запреты

- не хранить открытые пароли;
- не доверять клиентскому счёту;
- не обновлять счёт без judge lock;
- не редактировать finished match;
- не использовать wall clock/random напрямую в domain logic;
- не использовать `sleep` в тестах;
- не заменять integration test mock-тестом репозитория;
- не вводить background infrastructure без измеренной необходимости;
- не смешивать admin role с organizer/judge/captain.

## 6. Рекомендуемая структура модулей

- `auth`
- `admin`
- `users_profiles`
- `matches`
- `judging`
- `tournaments`
- `teams`
- `rankings_stats`
- `history`
- `notifications`
- `onboarding_help`
- `shared_kernel` — только действительно общие primitive types.

## 7. Первый запрос к Cursor

> Изучи весь пакет документов. Не пиши production-код. Сначала предложи архитектуру modular monolith, структуру репозитория, технологический стек, тестовую пирамиду и последовательность первых 15 TDD-задач. Для каждой задачи укажи requirement IDs, тесты Red phase, минимальную Green implementation и Definition of Done. Не добавляй внешние интеграции или функции вне MVP.
